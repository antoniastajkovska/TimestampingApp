## 1. Project Overview

A tamper-evident timestamping server that lets authenticated users prove a file existed at a given point in time. The system produces signed `.tsr` tokens (JWS RS256) and chains each record cryptographically to the previous one — any tampering breaks the hash chain.

Core security features:
- Email-based 2FA for both registration and login (8-digit OTP)
- Role-Based Access Control (RBAC) with hierarchical roles
- Just-In-Time (JIT) access elevation for USER_READER and USER_WRITER
- Tamper-evident log chain (SHA-256 hash of previous row)
- RSA-2048 JWS-signed timestamp tokens
- Browser-side SHA-256 hashing (file never leaves the client)
- PKI certificate chain for HTTPS + mutual TLS client authentication

## 5. Security Pillars

### 5.1 Authentication & 2FA

**Registration (2-step):**
1. `POST /api/auth/register` — validates uniqueness, BCrypt(12) hashes password, saves user as `enabled=false`, generates 8-digit OTP (15-min expiry), emails it.
2. `POST /api/auth/register/verify` — validates OTP (max 5 attempts, 15-min lockout on failure), sets `enabled=true`.

**Login (2-step):**
1. `POST /api/auth/login` — validates password via Spring Security `AuthenticationManager`, generates 8-digit OTP (10-min expiry), emails it. Supports login by username **or** email.
2. `POST /api/auth/verify-2fa` — validates OTP, creates an HttpOnly session cookie (`TSESSION`, `Secure`, `SameSite=Strict`).

**OTP Security:**
- 8-digit numeric codes (`SecureRandom`)
- Separate expiry times: 15 min (registration), 10 min (login)
- Max 5 attempts before 15-minute lockout (`totp_locked_until`)
- OTP cleared from DB after successful verification

**Registration retry:**
If a user starts registration but never completes OTP verification, the account stays in `enabled=false` state. Attempting to register again with the same username/email reuses the existing record (updates the password and issues a fresh OTP) rather than returning "Username already taken". Once an account is verified (`enabled=true`), the username and email are permanently reserved.

**Session Management:**
- In-memory `ConcurrentHashMap` store (no Redis required)
- 3600-second session lifetime
- `SessionFilter` extracts `TSESSION` cookie and populates `SecurityContext`
- Logout clears the session; password change invalidates all sessions for that user

**Password Policy (Passay):**
- Minimum 8 characters
- At least one uppercase, one lowercase, one digit, one special character

### 5.2 RBAC & JIT Access

**Role hierarchy:**

| Role | Type | Default | Permissions |
|------|------|---------|-------------|
| ORG_ADMIN | Organization | Assigned manually | View all users, manage users, create timestamps |
| ORG_USER | Organization | Assigned at registration | View own account |
| ORG_GUEST | Organization | — | Public view only |
| USER_READER | Resource (JIT) | Requested | View the registered users list |
| USER_WRITER | Resource (JIT) | Requested | Create timestamp entries |

**JIT Access Flow:**
- Any `ORG_USER` can request `USER_READER` (dashboard) or `USER_WRITER` (timestamp page)
- JIT grants expire after 15 minutes by default
- A `@Scheduled` task runs every 60 seconds to revoke expired JIT roles
- Live countdown timers shown in the UI (MM:SS)

**Permission Gates:**
- `GET /api/users` — requires `USER_READER` or `ORG_ADMIN`
- `POST /api/timestamp` — requires `USER_WRITER`
- `POST /api/users`, `PUT /api/users/{id}`, `DELETE /api/users/{id}` — requires `ORG_ADMIN`

**Admin CRUD (ORG_ADMIN only):**
- Create new users (pre-verified, assigned `ORG_USER` role)
- Edit any user's username/email
- Delete any user (cannot self-delete)

### 5.3 PKI Chain & Mutual TLS (Lab 4)

**Certificate chain produced by `generate-certs.sh`:**
```
FINKI CA (Root, 4096-bit, 10 years, self-signed)
    └── Lab CA (Intermediate, 4096-bit, 5 years)
          ├── Server cert (CN=localhost, 2048-bit, 1 year, SAN: DNS:localhost)
          └── Client cert (CN=TimestampingClient, 2048-bit, 1 year)
```

**Server TLS (Spring Boot `application.yml`):**
- `keystore.p12` — server cert chain (server → Lab CA → Root CA)
- `client-auth: need` — mutual TLS enabled; every HTTPS connection must present a valid client certificate signed by Lab CA or Root CA
- `truststore.p12` — Spring verifies client certs against Root CA and Lab CA

**Browser setup (one-time):**
1. Install `rootCA.crt` into the OS trust store (Windows: `certmgr.msc` → Trusted Root Certification Authorities → Import)
2. Import `client.p12` into Chrome (Settings → Privacy → Security → Manage certificates → Import; password: `ClientCert#Pw123`)

**Why mTLS?** Guarantees that only browsers/clients holding a certificate issued by the Lab CA can communicate with the server — network-level authentication before any application-level login.

### 5.4 RSA-Signed Timestamps & Log Chain

**Signing:**
- `CryptoService` loads an RSA-2048 key pair from PEM files at startup
- Each timestamp entry is signed as a JWS compact token (RS256 via Nimbus JOSE+JWT):
  ```json
  { "fileHash": "<sha256hex>", "seq": 42, "ts": "<ISO-8601>" }
  ```
- The JWS token is stored as the `signature` column in `log_chain`

**Log chain integrity:**
- Each row stores `SHA-256(canonical JSON of previous row)` as `previous_row_hash`
- First row uses `SHA-256("GENESIS")` as the sentinel
- Sequence numbers allocated from a PostgreSQL sequence (`log_chain_seq`) — no gaps, no races
- Row lock (`SELECT FOR UPDATE`) prevents concurrent entries from using the same previous hash

**Verification (`POST /api/timestamp/verify`, public endpoint):**
- Decodes the base64-encoded `.tsr` envelope
- Verifies JWS signature using the public key
- Checks that the `fileHash` claim matches the submitted file's hash
- Returns: valid/invalid, timestamp, sequence number, requestedBy

### 5.5 Browser-Side SHA-256 Hashing

The file is **never uploaded** to the server. The React frontend uses the Web Crypto API:

```javascript
// frontend/src/hooks/useFileSha256.js
const buffer = await file.arrayBuffer();
const hashBuf = await crypto.subtle.digest('SHA-256', buffer);
return Array.from(new Uint8Array(hashBuf))
  .map(b => b.toString(16).padStart(2, '0')).join('');
```

Only the 64-character hex digest is sent to `POST /api/timestamp`. This preserves user privacy — the server never sees the file content.

## 6. API Reference

All endpoints are under `https://localhost:8443/api`. Authentication via `TSESSION` HttpOnly cookie (set after successful 2FA).

### Auth endpoints (public)

| Method | Path | Body | Response |
|--------|------|------|----------|
| POST | `/auth/register` | `{username, email, password}` | `{pendingUsername}` |
| POST | `/auth/register/verify` | `{username, code}` | `204` |
| POST | `/auth/login` | `{username, password}` | `{pendingUsername}` |
| POST | `/auth/verify-2fa` | `{username, code}` | Sets `TSESSION` cookie |
| POST | `/auth/logout` | — | Clears cookie |

### User endpoints (authenticated)

| Method | Path | Auth | Body | Response |
|--------|------|------|------|----------|
| GET | `/users/me` | Any | — | `UserProfileResponse` |
| PUT | `/users/me` | Any | `{username, email, firstName, lastName}` | `UserProfileResponse` |
| PUT | `/users/me/password` | Any | `{currentPassword, newPassword, confirmPassword}` | `{message}` |
| DELETE | `/users/me` | Any | `{password}` | `{message}` |
| GET | `/users` | USER_READER or ORG_ADMIN | — | `[UserSummary]` |
| POST | `/users` | ORG_ADMIN | `{username, email, password}` | `UserSummary` |
| PUT | `/users/{id}` | ORG_ADMIN | `{username, email}` | `UserSummary` |
| DELETE | `/users/{id}` | ORG_ADMIN | — | `{message}` |

**UserProfileResponse** fields: `id, username, email, firstName, lastName, createdAt, updatedAt, writerExpiresAt, readerExpiresAt, roles`

### JIT endpoints (authenticated)

| Method | Path | Response |
|--------|------|----------|
| POST | `/jit/request-writer` | `{message, expiresAt}` — grants USER_WRITER for 15 min |
| POST | `/jit/request-reader` | `{message, expiresAt}` — grants USER_READER for 15 min |

### Timestamp endpoints

| Method | Path | Auth | Body | Response |
|--------|------|------|------|----------|
| POST | `/timestamp` | USER_WRITER | `{fileHash}` | `TimestampResponse {id, sequenceNumber, fileHash, timestampUtc, signature, previousRowHash}` |
| POST | `/timestamp/verify` | Public | `{token, fileHash}` | `VerifyResponse {valid, timestamp, sequenceNumber, requestedBy, message}` |

## 8. Running the Application

### Prerequisites
- Java 21
- Maven 3.9+
- PostgreSQL 15 (database: `timestampingdb`)
- Node.js 20+
- OpenSSL 3.x + Java `keytool` (for certificate generation)
- Gmail App Password (for OTP emails)

### One-time setup

**1. Generate the full PKI chain + keystores:**
```bash
cd TimestampingApp/backend
bash ../generate-certs.sh
```
This creates the Root CA → Lab CA → server cert + client cert chain and packages them into `keystore.p12` (server TLS) and `truststore.p12` (mTLS).

**2. Install Root CA into your OS trust store (Windows):**
```
certmgr.msc → Trusted Root Certification Authorities → Import → rootCA.crt
```

**3. Install client certificate into Chrome:**
```
Settings → Privacy and security → Security → Manage certificates → Import
File: backend/src/main/resources/certs/client.p12
Password: ClientCert#Pw123
```

**4. Create the PostgreSQL database:**
```sql
CREATE DATABASE timestampingdb;
```

**5. Run the DB migration (if upgrading from lab2):**
```bash
"C:\Program Files\PostgreSQL\18\bin\psql.exe" -U postgres -d timestampingdb -f migrate_lab3.sql
```

**6. Install frontend dependencies:**
```bash
cd frontend
npm install
```

### Gmail SMTP setup (one-time)

OTP emails are sent via Gmail SMTP. Plain Gmail passwords are rejected — you must create an **App Password**:

1. Sign in to the sender Gmail account at [myaccount.google.com](https://myaccount.google.com)
2. Security → **2-Step Verification** → Enable
3. Security → **App passwords** → App: Mail, Device: Other (`TimestampingApp`) → **Generate**
4. Copy the 16-character password (spaces are ignored — remove them)
5. Set `MAIL_USER` and `MAIL_PASS` in `backend/.env` (see below)

The credentials are already baked into `application.yml` defaults for the project Gmail account (`timestampingapp.dev@gmail.com`).

### `.env` file (`backend/.env`)

```env
DB_USER=postgres
DB_PASSWORD=postgres

MAIL_USER=timestampingapp.dev@gmail.com
MAIL_PASS=cjbfpciwvpnbmsqm        # Gmail App Password

SSL_KEYSTORE_PASSWORD=changeit
PRIVATE_KEY_PATH=src/main/resources/certs/private.pem
PUBLIC_KEY_PATH=src/main/resources/certs/public.pem

# Uncomment to enable mTLS after running generate-certs.sh:
# SSL_CLIENT_AUTH=need
# SSL_TRUSTSTORE_PATH=classpath:certs/truststore.p12
# SSL_TRUSTSTORE_PASSWORD=changeit
```

Load it in Git Bash before starting:
```bash
export $(grep -v '^#' .env | xargs)
```

### Starting the servers

**Backend (terminal 1):**
```bash
cd backend
export $(grep -v '^#' .env | xargs)
mvn spring-boot:run
```

**Frontend (terminal 2):**
```bash
cd frontend
npm start
# opens https://localhost:3001
```
